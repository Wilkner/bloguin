📄 RFC 002: Arquitetura e Especificação do Sistema "Bloguin"
Status: PRONTO PARA IMPLEMENTAÇÃO

Autor: Wilkner

Data: Julho de 2026

1. Objetivos do Produto (Propósito)
O Bloguin é uma plataforma de crônicas de viagem de alta performance baseada no conceito Jamstack de entrega de conteúdo estático rápido, apoiada por uma API privada de administração e enriquecimento de dados.

Objetivos Principais:
Consumo Multimodal: Permitir que o visitante consuma a mesma história em formato de Fotos com Legenda (estilo Stories), Artigo Detalhado (Texto Literário) ou Vídeo Premium (Narração/Avatar).

Administração Automatizada: Criar um fluxo de publicação onde o autor faz o upload do relato e da imagem da viagem, delegando ao backend a extração automática de metadados reais da fotografia via cabeçalho EXIF.

Engajamento Serverless: Fornecer um sistema reativo de curtidas/reações utilizando a infraestrutura do Supabase sem onerar o servidor de hospedagem.

2. Arquitetura Geral do Ecossistema
O ecossistema é dividido em duas frentes independentes para garantir custo zero de escala e isolamento de segurança:

Plaintext
[ FRONTEND ESTÁTICO ] ──(Hospedado no GitHub Pages)──> Lê arquivos .txt locais e interage com Supabase
[ BACKEND ADMIN ]      ──(Hospedado no Render)────────> Painel Privado, Extração EXIF e Grita arquivos p/ o Repo
[ PERSISTÊNCIA ]       ──(Supabase PostgreSQL)────────> Guarda os metadados de catálogo e as reações em real-time
Camada de Entrega (Client): Single Page Application (SPA) construída em HTML5/Vanilla JS e CSS customizado hospedada no GitHub Pages. Consome os arquivos de texto diretamente da pasta local via requisições HTTP assíncronas.

Camada de Controle e Enriquecimento (Admin API): API construída em Python (FastAPI/Flask) hospedada no Render. Seu uso é estritamente administrativo para processamento de arquivos pesados, uploads e parsing.

4. Engenharia de Dados e Contratos de Interface
4.1. O Padrão de Armazenamento Literário (Front Matter)
As histórias devem ser persistidas na pasta /cronicas/ como arquivos de texto (.txt) estruturados com cabeçalho de metadados isolado por três hífens (---). O parser do front-end fará a leitura isolada desse bloco.

Plaintext
---
date: 2026-07-16
destination: Itália
image: assets/images/david_01.jpg
video: assets/videos/david_michelangelo.mp4
camera: Apple iPhone 15 Pro
latitude: 43.7769
longitude: 11.2587
legenda: Olhando de perto na Galleria dell'Accademia, as veias parecem pulsar.
---

4.2. Modelagem Relacional no Supabase (PostgreSQL)
SQL
-- Catálogo de crônicas para renderização automática do grid inicial
CREATE TABLE public.cronicas (
    id TEXT PRIMARY KEY,                 -- Ex: 'david_michelangelo'
    titulo TEXT NOT NULL,
    destino TEXT NOT NULL,
    capa_url TEXT,                       -- Aponta para assets/images/
    arquivo_path TEXT NOT NULL,          -- Aponta para cronicas/
    criado_em TIMESTAMPTZ DEFAULT NOW()
);

-- Tabela de contadores atômicos para engajamento por emoji
CREATE TABLE public.reacoes (
    cronica_id TEXT REFERENCES public.cronicas(id) ON DELETE CASCADE,
    emoji TEXT NOT NULL,                 -- '🔥', '❤️', '😂'
    contador INT DEFAULT 0,
    PRIMARY KEY (cronica_id, emoji)
);
5. Especificação do Fluxo Administrativo (O Motor EXIF)
Quando o formulário administrativo submeter uma nova história, o backend executará a seguinte esteira lógica em pipeline:

Persistência da Mídia: A imagem enviada pelo formulário é salva em assets/images/ recebendo o ID sanitizado (substituindo hífens por sublinhados: _).

Varredura EXIF: A biblioteca Pillow lê os metadados binários da imagem procurando pelas tags nativas:

DateTimeOriginal ➡️ Convertido de AAAA:MM:DD para AAAA-MM-DD.

Model ➡️ Mapeia o dispositivo que capturou a imagem.

GPSInfo ➡️ Converte graus, minutos e segundos binários em coordenadas geográficas decimais puras (Latitude/Longitude).

Geração e Output: O Python mescla esses dados extraídos com o texto inserido no formulário, injeta o Front Matter e grava o arquivo final .txt na pasta correspondente.
